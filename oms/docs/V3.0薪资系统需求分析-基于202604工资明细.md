# 薪资系统需求分析说明（最终口径版）

基于文件：`202604工资明细.xlsx`

复核日期：2026-05-28

## 1. 总体结论

这份 Excel 不是一张普通工资表，而是一套用 Excel 搭起来的薪资结算系统。

它已经包含：

1. 总工资表。
2. 分公司工资明细。
3. 社保明细。
4. 公积金明细。
5. 个税申报计算。
6. 钉钉工资卡。
7. 月人力成本汇总。
8. 考勤汇总。
9. 劳务费明细。

所以后续如果开发系统，不能只做“工资录入表”，而要做“薪资核算系统”。系统要把工资、考勤、绩效、社保、公积金、个税、工资条、成本汇总这些流程连起来。

## 2. 发薪和月份口径

公司当前发薪规则是：

```text
每月 26 号发上月薪资。
```

也就是：

```text
次月 26 号发上月工资。
```

例如 `202604工资明细.xlsx` 的意思是：

```text
发的是 2026 年 04 月薪资。
```

具体口径如下：

| 项目 | 口径 |
|---|---|
| 工资归属月 | 2026 年 04 月 |
| 考勤月份 | 2026 年 04 月 |
| 绩效月份 | 2026 年 04 月 |
| 发放日期 | 通常为 2026 年 05 月 26 日 |
| 发放月 | 2026 年 05 月 |
| 社保公积金缴费月 | 正常情况下是 2026 年 05 月 |
| 个税所属期 | 2026 年 05 月 |

系统里不能只设置一个“工资月份”。至少要区分：

1. 工资归属月。
2. 发放日期。
3. 发放月。
4. 社保公积金缴费月。
5. 个税所属期。

系统页面建议展示为：

```text
工资归属月：2026-04
考勤月份：2026-04
绩效月份：2026-04
发放日期：2026-05-26
社保公积金缴费月：2026-05
个税所属期：2026-05
```

## 3. 现有 Excel 的计算流程

现有 Excel 的计算流程可以简单理解为：

1. 先在 `202604工资表` 里维护员工、公司、起薪时间、月薪基数、岗位、绩效等级、补发扣。
2. 总工资表根据岗位和绩效计算扣除前工资。
3. 再按公司拆成尚诚能源、南大尚诚、基石三套工资明细。
4. 工资明细再关联社保、公积金、个税数据。
5. 工资明细算出员工最终实发工资。
6. 钉钉工资卡整理员工可见的工资条。
7. 月成本汇总统计公司人力成本。
8. 考勤表记录当月考勤，考勤异常进入补扣款。
9. 马老师劳务费单独计算，不进入工资薪金累计个税。

## 4. 几个工资字段必须分清

### 4.1 总工资表里的“实际应发薪资”

Excel 里的“实际应发薪资”不是员工最终到手工资。

它的逻辑是：

```text
实际应发薪资 = 应发薪资 + 补发扣
应发薪资 = 基本薪资 + 绩效薪资
```

它还没有扣个人社保、个人公积金和个税。

为了系统里更清楚，建议理解为：

```text
扣除前应发工资
```

如果为了和 Excel 保持一致，页面上仍然叫“实际应发薪资”，也要在字段说明里写清楚：它不是最终实发。

### 4.2 工资明细里的“实发工资”

工资明细里的“实发工资”才是员工最终到手金额。

公式是：

```text
实发工资 = 扣除前应发工资 - 个人社保 - 个人公积金 - 本月个税
```

### 4.3 钉钉工资卡里的“实发金额”

钉钉工资卡里的“实发金额”也表示员工最终到手金额。

它应和工资明细里的“实发工资”一致。

## 5. 总工资表规则

总工资表是工资核算的起点。

主要字段包括：

1. 姓名。
2. 公司。
3. 起薪时间。
4. 花名册薪资。
5. 月薪基数。
6. 月薪资总额。
7. 岗位。
8. 基本薪资。
9. 绩效薪资基数。
10. 绩效等级。
11. 绩效薪资。
12. 应发薪资。
13. 补发扣。
14. 实际应发薪资。

### 5.1 岗位拆分规则

当前 Excel 规则如下：

| 岗位/人员类型 | 基本薪资比例 | 绩效薪资基数比例 |
|---|---:|---:|
| 南京大学 | 100% | 0% |
| BOSS | 100% | 0% |
| 实习生 | 100% | 0% |
| 人力劳务 | 100% | 0% |
| 部门负责人 | 60% | 40% |
| 员工 | 70% | 30% |

公式：

```text
基本薪资 = 月薪资总额 × 基本薪资比例
绩效薪资基数 = 月薪资总额 × 绩效薪资比例
```

### 5.2 绩效规则

`202604` 工资固定使用 `202604` 绩效等级，不延后到下月。

当前 Excel 绩效系数如下：

| 绩效等级 | 系数 |
|---|---:|
| A | 1.5 |
| B | 1.2 |
| C | 1.0 |
| D | 0.8 |
| E | 0.6 |
| 无等级 | 0 |

公式：

```text
绩效薪资 = 绩效薪资基数 × 绩效系数
```

这些规则后续要做成系统配置，不要写死。

## 6. 公司和内部归属

Excel 里的公司字段会影响工资明细、社保、公积金、个税和成本汇总。

其中 `南大尚诚（人力）` 不是独立公司。它本质上仍然是 `南大尚诚`，只是内部人员归属提醒，表示这部分人是南大尚诚的劳务人力。

系统建议分成两个字段：

| 字段 | 说明 |
|---|---|
| 薪资结算公司 | 真正参与工资、社保、公积金、个税和成本归集的公司，例如南大尚诚 |
| 内部归属 | 公司内部人员分类，例如南大尚诚（人力） |

这样汇总公司数据时，`南大尚诚（人力）` 应并入 `南大尚诚`。

## 7. 公司调转规则

员工公司调转时，工资、考勤、绩效按工资归属月判断。

### 7.1 1 号调转

如果员工 `2026-04-01` 从南大尚诚调到能源，那么：

```text
202604 工资由能源发。
202604 考勤看 4 月。
202604 绩效看 4 月。
```

原因是员工 4 月 1 日已经是能源的人，所以 4 月工资肯定由能源发。

### 7.2 非 1 号调转

目前业务中暂不存在非 1 号调转。

如果以后出现非 1 号调转，当前口径先按简单规则处理：

```text
社保公积金归前一个公司。
```

这个场景一期可以不做复杂拆分，但系统应保留手工调整能力。

## 8. 社保公积金扣款规则

这是系统最重要的规则之一。

核心原则：

```text
社保公积金在哪个公司产生，就必须在发这个公司的薪资时扣除。
```

不能只看发薪月份，也不能随便扣到其他公司的工资里。

### 8.1 正常情况

正常员工发 `202604` 工资时，通常扣 `202605` 社保公积金。

因为：

```text
202604 工资在 202605 发放。
202605 产生社保公积金费用。
```

### 8.2 离职情况

如果员工 4 月离职，5 月不缴社保公积金，那么发 4 月工资时不扣社保公积金。

示例：

```text
员工 2026 年 04 月离职
2026 年 05 月不产生社保公积金
2026-05-26 发 202604 工资时，不扣 202605 社保公积金
```

### 8.3 入职情况

入职当月的工资发放时，会一次性扣入职月和发放月两个月社保公积金。

示例：

```text
员工 2026-03-01 入职
2026-04-26 发 202603 工资
一次性扣 202603 和 202604 两个月社保公积金
```

### 8.4 公司调转情况

社保公积金必须扣在产生公司的工资里。

示例：

```text
员工 2026-04-01 从南大尚诚调到能源
202604 工资由能源发
202605 社保公积金也从能源产生
```

这种情况下，`2026-05-26` 发 `202604` 工资时，不扣 `202605` 社保公积金。

等到 `2026-06-26` 发 `202605` 工资时，再一次性扣：

```text
202605 社保公积金
+ 202606 社保公积金
```

原因是：这两个月社保公积金都是能源产生的，所以要在能源后续工资里扣除，不能扣到其他公司的工资里。

如果出现某笔社保公积金已经在某公司产生，但当月工资没有扣，则这笔金额要形成待扣记录，等发该公司薪资时扣。

系统应记录每笔社保公积金的产生公司和扣款工资：

| 字段 | 说明 |
|---|---|
| 员工 | 员工编号和姓名 |
| 社保公积金产生公司 | 这笔社保公积金在哪家公司产生 |
| 缴费月 | 社保公积金对应月份 |
| 个人社保金额 | 员工个人承担社保 |
| 个人公积金金额 | 员工个人承担公积金 |
| 是否已扣 | 是否已在工资中扣除 |
| 扣在哪个月工资 | 实际在哪个工资归属月扣除 |
| 扣款原因 | 正常扣款、入职补扣、调转补扣、离职停扣等 |

## 9. 社保计算规则

社保分为公司承担和个人承担。

### 9.1 公司承担

当前 Excel 主要规则：

| 项目 | 公式 |
|---|---|
| 单位养老 | 缴费基数 × 16% |
| 单位工伤 | 缴费基数 × 工伤比例 |
| 单位失业 | 缴费基数 × 0.5% |
| 单位医疗 | 缴费基数 × 7% |
| 单位生育 | 缴费基数 × 0.8% |
| 单位合计 | 上述项目合计 |

工伤比例不同公司可能不同，样表里有 0.32%、0.2% 等情况，所以必须配置。

### 9.2 个人承担

当前 Excel 主要规则：

| 项目 | 公式 |
|---|---|
| 个人养老 | 缴费基数 × 8% |
| 个人失业 | 缴费基数 × 0.5% |
| 个人医疗 | 缴费基数 × 2% |
| 个人大病 | 固定 10 元 |
| 个人合计 | 上述项目合计 |

社保表中显示了上限 24762 元/月、下限 4952 元/月。系统要把上下限做成配置。

## 10. 公积金计算规则

样表中确认到：

| 公司 | 单位比例 | 个人比例 |
|---|---:|---:|
| 尚诚能源 | 10% | 10% |
| 南大尚诚 | 10% | 10% |
| 基石 | 12% | 12% |

公式：

```text
单位公积金 = 公积金基数 × 单位比例
个人公积金 = 公积金基数 × 个人比例
公积金合计 = 单位公积金 + 个人公积金
```

当前 Excel 使用 `ROUND(基数 × 比例, 0)`，也就是取整数。

系统要支持按公司和月份配置比例，也要支持配置取整规则。

## 11. 个税规则

个税口径已经确认：

```text
202604 工资在 202605 申报。
```

也就是工资归属月是 2026 年 04 月，但个税所属期是 2026 年 05 月。

当前工资薪金个税按累计预扣法计算。

通俗理解：

```text
不是只看本月工资算税，而是看今年累计收入、累计扣除、累计已扣税。
```

核心逻辑：

```text
累计收入 = 上月累计收入 + 本月扣除前应发工资
累计扣除 = 上月累计扣除 + 本月个人社保 + 本月个人公积金 + 本月专项附加扣除
累计应纳税所得额 = 累计收入 - 累计扣除 - 累计减除费用
累计应纳税额 = 累计应纳税所得额 × 税率 - 速算扣除
本月个税 = 累计应纳税额 - 上月累计已扣个税
```

如果累计应纳税所得额小于 0，按 0 处理。  
如果本月个税小于 0，按 0 处理。

### 11.1 个税税率表

| 累计应纳税所得额 | 税率 | 速算扣除 |
|---:|---:|---:|
| 0 - 36000 | 3% | 0 |
| 36000 - 144000 | 10% | 2520 |
| 144000 - 300000 | 20% | 16920 |
| 300000 - 420000 | 25% | 31920 |
| 420000 - 660000 | 30% | 52920 |
| 660000 - 960000 | 35% | 85920 |
| 960000 以上 | 45% | 181920 |

### 11.2 上月累计数据

目前上月累计数据是从上月 Excel 复制来的。

后续系统要自己保存每个月的累计数据，包括：

1. 上月累计收入。
2. 上月累计扣除。
3. 上月累计个税。
4. 上月累计减除费用。

如果后续提供个税系统导出的记录，系统可以用于核对。

核对原则：

```text
以上月工资表累计数据为准，再和个税系统导出数据核对差异。
```

## 12. 考勤和补发扣规则

`202604` 工资看 `202604` 考勤。

病假、事假、旷工等已经体现在 Excel 的考勤里。

后续系统里，补发扣先分两类：

1. 考勤。
2. 其他。

考勤补扣应该从考勤结果来，其他补扣由人工录入并填写原因。

如果后续要做更细，可以再拆：

1. 病假扣款。
2. 事假扣款。
3. 旷工扣款。
4. 补发工资。
5. 社保公积金调整。
6. 其他调整。

但一期先分“考勤”和“其他”即可。

## 13. 负数实发规则

如果因为入职、调转等原因补扣多个月社保公积金，导致实发工资为负数，系统先允许出现负数。

原因是：

```text
出现负数后，可以清楚看到还差多少钱。
```

系统不要自动把实发工资截断为 0。

但要做明显提醒：

```text
实发工资为负数，请确认扣款是否正确。
```

## 14. 劳务费规则

`马骏老师202604` 是劳务费，不是正式员工工资。

规则已经确认：

1. 马老师劳务费每月都有。
2. 每月固定 8000。
3. 按自然人劳务报酬计算。
4. 不计入工资薪金累计个税。
5. 可以进入月人力成本汇总。

所以系统里要单独做“劳务费”或“劳务人员结算”，不要混进正式员工工资薪金。

## 15. 钉钉工资卡规则

钉钉工资卡是员工工资条。

它不是单纯展示标准工资，而是要展示“本次发薪实际发了什么、实际扣了什么”。

当前字段包括：

1. 姓名。
2. 基本工资。
3. 绩效薪资基数。
4. 绩效等级。
5. 绩效薪资。
6. 加班薪资。
7. 考勤补扣。
8. 其他补扣。
9. 实际应发工资。
10. 社保合计。
11. 住房公积金。
12. 个人所得税。
13. 实发金额。

公式：

```text
实际应发工资 = 基本工资 + 绩效薪资 + 加班薪资 + 考勤补扣 + 其他补扣
实发金额 = 实际应发工资 - 社保合计 - 住房公积金 - 个人所得税
```

工资条里的实发金额要和工资明细里的实发工资一致。

如果员工存在入职、离职、公司调转等情况，工资条里必须把社保公积金扣款说明清楚。

例如：

1. 正常扣 1 个月社保公积金：显示本次扣款月份和金额。
2. 入职首月扣 2 个月社保公积金：显示扣了哪两个月，例如 `202603 + 202604`。
3. 公司调转后暂时不扣社保公积金：工资条里显示本次未扣社保公积金，并注明原因。
4. 后续补扣 2 个月社保公积金：工资条里显示补扣月份，例如 `202605 + 202606`。

系统里的工资条建议增加“扣款明细说明”，不要只显示一个社保合计数字。

建议展示为：

| 项目 | 示例 |
|---|---|
| 社保扣款月份 | 202605、202606 |
| 公积金扣款月份 | 202605、202606 |
| 社保产生公司 | 尚诚能源 |
| 公积金产生公司 | 尚诚能源 |
| 扣款类型 | 正常扣款、入职补扣、调转补扣、离职停扣 |
| 备注 | 本次补扣 2 个月社保公积金 |

## 16. 月成本汇总规则

月成本汇总表容易被误解，要写清楚。

它的逻辑是：

```text
合计 = 实发工资 + 个人社保 + 个人公积金 + 个税 + 公司社保 + 公司公积金 + 其他
```

这不是重复计算。

因为：

```text
实发工资 + 个人社保 + 个人公积金 + 个税 = 扣除前应发工资
```

所以成本汇总本质是：

```text
公司总成本 = 扣除前应发工资 + 公司承担社保 + 公司承担公积金 + 其他费用
```

月成本汇总还要包含：

1. 正式员工工资成本。
2. 公司承担社保。
3. 公司承担公积金。
4. 劳务费。
5. 蚂蚁社保等特殊成本。
6. 其他费用。

### 16.1 调公司时成本必须按公司拆开

月成本汇总是给公司做成本归集用的，所以不能只按员工汇总一个总数。

如果员工发生公司调转，系统要把该员工在不同公司产生的费用分别列清楚。

例如某员工从南大尚诚调到尚诚能源，系统成本汇总里要能看到：

| 员工 | 成本产生公司 | 成本月份 | 成本项目 | 金额 |
|---|---|---|---|---:|
| 张三 | 南大尚诚 | 202604 | 工资/社保/公积金/个税/其他 | 按实际金额 |
| 张三 | 尚诚能源 | 202605 | 工资/社保/公积金/个税/其他 | 按实际金额 |

也就是说：

```text
工资条关注“本次员工实际发扣”。
成本汇总关注“费用在哪家公司产生”。
```

这两个口径不能混。

系统成本汇总至少要支持以下维度：

1. 员工。
2. 成本产生公司。
3. 内部归属。
4. 工资归属月。
5. 社保公积金缴费月。
6. 费用类型。
7. 金额。

费用类型至少包括：

1. 扣除前应发工资。
2. 公司承担社保。
3. 公司承担公积金。
4. 个人社保。
5. 个人公积金。
6. 个税。
7. 劳务费。

如果同一个员工同一个月涉及两个公司，系统要能分别列出两家公司发生的费用，再提供员工合计和公司合计。

## 17. 当前 Excel 风险

现有 Excel 能跑，但长期风险比较明显：

1. 很多公式按姓名匹配，重名、错字、空格都会有风险。
2. 很多表靠表名和列号引用，表名改了或列位置变了就容易错。
3. 个税依赖上月累计数据，一旦漏复制或取错，本月个税就会错。
4. 社保公积金涉及入职、离职、调转，不能只靠发薪月简单判断。
5. 审批、修改、导出缺少系统留痕。
6. 身份证号、工资、银行卡等敏感信息需要权限控制。

系统必须使用员工编号作为唯一标识，姓名只做展示。

## 20. 审批流程

建议流程：

1. 人事生成工资表。
2. 人事核对考勤、绩效、补扣款。
3. 财务核对社保、公积金、个税。
4. 财务核对劳务费和成本汇总。
5. 管理层审批。
6. 审批通过后锁定。
7. 导出工资表、工资条、发放表、成本汇总。
8. 标记已发放。
9. 归档。

已审批后不能随便改。确实要改，必须走更正流程，并记录原因。

## 21. 系统必须做的校验

1. 员工编号是否缺失。
2. 同一工资月员工是否重复。
3. 员工是否有有效薪资档案。
4. 岗位是否配置拆分规则。
5. 绩效等级是否配置系数。
6. 花名册薪资和月薪基数是否不一致。
7. 月薪基数和月薪资总额是否不一致。
8. 补发扣是否有分类和原因。
9. 社保公积金产生公司和工资结算公司是否一致。
10. 入职补扣月份是否正确。
11. 离职员工是否错误扣了发放月社保公积金。
12. 调转员工社保公积金是否扣在正确公司。
13. 公积金比例是否按公司取对。
14. 社保比例和上下限是否取对。
15. 上月累计收入是否缺失。
16. 上月累计扣除是否缺失。
17. 上月累计个税是否缺失。
18. 个税所属期是否和发放月一致。
19. 工资条实发金额和工资明细实发工资是否一致。
20. 工资条是否展示社保公积金扣款月份、产生公司和扣款原因。
21. 实发工资为负数时是否提醒。
22. 成本汇总是否按成本产生公司拆分。
23. 成本汇总是否和明细合计一致。
24. 劳务费是否未混入工资薪金累计个税。

## 22. 研发验收标准

一期上线前，必须用这份脱敏 Excel 做并行核算。

验收标准：

1. 总工资表的基本薪资、绩效薪资、扣除前应发工资和 Excel 一致。
2. 考勤补扣和其他补扣分类正确。
3. 社保个人部分和公司部分和 Excel 一致。
4. 公积金个人部分和公司部分和 Excel 一致。
5. 入职补扣两个月社保公积金逻辑正确。
6. 离职后发上月工资不扣发放月社保公积金。
7. 公司调转后，社保公积金扣在产生公司的工资里。
8. 个税所属期和发放月一致。
9. 本月个税和 Excel 一致。
10. 最终实发工资和 Excel 一致。
11. 允许实发工资为负数，并有明显提醒。
12. 钉钉工资卡实发金额和工资明细一致。
13. 钉钉工资卡能体现本次未扣、补扣 2 个月等社保公积金特殊情况。
14. 马老师劳务费按劳务报酬处理，不进入工资薪金累计个税。
15. 月成本汇总和 Excel 一致。
16. 调公司员工在南大尚诚、尚诚能源等不同公司产生的费用能分别列清。
17. 审批后工资表锁定。
18. 所有导入、修改、审批、导出都有日志。

## 23. 仍需业务确认的少量问题

目前大规则已经清楚，只剩少量细节以后开发前确认即可：

1. 非 1 号调转如果未来发生，工资是否需要拆分，还是只做人工调整。
2. 考勤补扣的具体金额公式是否要系统自动算，还是先人工录入。
3. “其他补扣”是否还要细分更多类型。
4. 个税系统导出表的固定格式是什么。
5. 银行代发文件是否一期就要做。

## 24. 最终建议

系统建设建议按三步走：

1. 一期：替代当前 Excel 薪资核算闭环，保证工资、考勤、绩效、社保、公积金、个税、劳务费、工资条、成本汇总都能准确对账。
2. 二期：接入钉钉考勤、钉钉工资条、银行代发、个税申报等外部系统。
3. 三期：做人力成本分析、薪酬分析和预算预测。

这版需求的核心原则是：

```text
先把现有 Excel 的逻辑准确系统化，不急着做大而全。
先保证工资算得准、扣得对、查得到、审得过、追得回。
```
