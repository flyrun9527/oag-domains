# OAG 本体建模指南

本文档说明如何把业务文档建模为 OAG `ontology.yaml`。它不是字段规范；字段语义见
[metamodel-spec.md](metamodel-spec.md)。这里关注的是建模判断：哪些内容应该成为对象，
哪些应该成为函数、规则、工作流，哪些应该下沉为确定性校验。

核心原则是：本体不是文档摘要，而是 Agent 执行业务任务时可依赖的结构化执行边界。
文档负责解释业务，本体负责让模型能够正确选择工具，并在关键步骤上不轻易走偏。

## 从任务闭环开始

不要先按文档名词列表建模。先找出用户会让 Agent 完成的任务闭环：

- 任务从哪里开始：申请、事件、预警、订单、工单、案件。
- 中间需要哪些判断：等级、规则、约束、合规、审批、复核。
- 会产生哪些过程产物：检查记录、候选方案、调度单、审批记录、校验结果。
- 任务如何结束：最终方案、报告、处置结论、无解结论、状态关闭。

例如，高压接入领域的闭环更像：

```text
申请 -> 资源筛选 -> 救济/补充 -> 方案组合 -> 评分 -> 最终输出
```

公路应急与无人机领域的闭环可能是：

```text
事件 -> 影响设施 -> 检查评估 -> 响应等级 -> 抢通/调度/管制 -> 通行评估 -> 报告
侦测任务 -> 合规检查 -> 飞行审批 -> 调派执行 -> 数据采集
```

先确定闭环，再决定 `objects`、`functions`、`rules` 和 `workflows`。

## 对象分层

建模时可以把对象分成五类。一个领域不一定五类都有，但分类有助于控制本体边界。

| 类型 | 含义 | 典型例子 |
|---|---|---|
| 事实对象 | 外部系统或静态数据中的业务实体 | 路段、桥梁、无人机、变电站、馈线、客户 |
| 规则对象 | 人工确认的规则表、映射表、参数表 | 损伤分级标准、响应级别规则、电源要求、审批规则 |
| 入口对象 | 启动任务的请求、事件或预警 | 接入申请、增容申请、灾害事件、交通事故、气象预警 |
| 过程产物对象 | Agent 执行过程中生成的中间记录 | 检查记录、合规检查、划接记录、审批记录、候选方案 |
| 终态输出对象 | 可交付给用户或外部系统的结果 | 最终方案、事件报告、绕行方案、通行评估、无解结论 |

事实对象通常是业务世界中可被读取和引用的稳定实体；规则对象通常是人工确认的规则表、映射表或参数表；过程产物和候选结果通常只存在于一次任务执行过程中。

### 事实对象优先

事实对象是本体的业务骨架。它们不是文档里的普通名词，也不是某个流程步骤的临时视图，而是业务世界中相对稳定、可以被任务反复读取、引用和关联的实体。

建模时应先识别事实对象，再识别过程产物。判断一个概念是否应成为事实对象，可以问：

- 它是否在多条任务闭环中反复出现，而不是只属于某一步？
- 它是否有稳定标识，例如编号、代码、坐标、状态、所属关系或生命周期？
- Agent 是否需要围绕它做查询、过滤、关联、容量/状态/合规判断？
- 它是否是业务人员会自然识别和追踪的对象，而不是某次推理临时拼出的视图？

如果答案大多为“是”，优先建成事实对象。即使当前尚未确定数据来源，也不要为了贴合某一步流程而把它退化成抽象流程对象。数据来源和字段补齐可以在本体初稿之后由开发人员根据实际系统调整。

反过来，下面这些通常不应建成事实对象：

- 某次搜索返回的结果集合。
- 某一步校验得到的通过/失败记录。
- 某轮推理中的候选、评分、排序、推荐。
- 为了方便描述流程而合成的“能力”“视图”“批次”。

这些更适合建成过程产物对象或终态输出对象，并标记为 `agent_generated`。如果一个搜索结果视图确实需要保留，也应明确它依附于哪些事实对象，而不是替代事实对象本身。

## 建模关口

业务文档中的这些表达通常意味着需要显式建模为关口对象、关口函数或确定性校验：

- 必须、不得、至少、优先。
- 通过后才能、不通过则、若不满足则。
- 审批后、确认后、复核后、验收后。
- 超时、升级、转派、终止、回退。
- 双回路、双电源、合规、禁飞、禁止通行等强约束。

关口不要只写在 prompt 里。更稳的表达方式包括：

- 建成中间对象，例如 `ComplianceCheck`、`FlightApproval`、`TransferVerification`。
- 建成业务函数，例如 `check_compliance`、`validate_source_requirement`。
- 在函数 handler 或 validator 中返回结构化错误和修正提示。
- 用 `preconditions`、`effects`、`status_transitions` 表达可执行边界。

Prompt 可以导航流程，但不能作为正确性的唯一防线。

## 函数粒度

函数不是越多越好。判断一个函数是否值得暴露给 Agent，可以看它是否承载了业务语义。

适合保留为领域函数：

- 跨对象查询或聚合：按事件查影响设施、跨申请类型查请求。
- 空间、范围、状态过滤：查附近可用无人机、查半径内接入点。
- 规则口径封装：查电源要求、查审批要求、查损伤等级。
- 业务动作：检查、审批、调度、划接、生成方案、评分、报告。
- 关键关口：合规检查、复核、校验、无解判断。

通常不必保留为领域函数：

- 只是 `query(Object, id)` 的薄包装。
- 只是把对象字段原样返回、没有聚合或业务解释的 getter。
- 与通用工具能力重复、且会增加模型选工具噪音的函数。

这个规则不是绝对的。如果一个 `get_xxx` 封装了跨对象查找、空间搜索、可用性过滤或业务口径，
它仍然值得保留。

## 规则与函数的边界

能表格化、枚举化、分类化的判断，优先放到 `rules` 或规则对象中：

- 等级映射。
- 分类规则。
- 状态到处置建议。
- 固定权重评分。
- 审批类型和提前量。

需要读取多个对象、生成记录、修改状态、执行搜索或包含算法过程的逻辑，放到 `functions`。

当规则来自法规、制度或人工确认材料时，应尽量保留来源字段或说明，方便审计和后续复核。

## Workflow 的角色

`workflows` 适合表达标准业务流程和分支导航，例如：

```text
任务规划 -> 合规检查 -> 飞行审批 -> 调派执行 -> 数据采集
```

但 workflow 不是完整的业务正确性保证。稳定执行通常需要三层配合：

- workflow 说明大方向和步骤顺序。
- function 的 `usage_prompt`、`preconditions` 和 `effects` 说明调用条件和副作用。
- handler、validator 和 stop check 在模型走错时返回结构化错误或阻止错误结论。

如果某一步只是 mock 或占位实现，应在领域状态中明确标记，避免把流程图误认为真实执行能力。

## 数据来源与可变性

`data_source` 和 `mutability` 不只是文档标签，也会影响运行时策略和确认边界。

- `external_api + read_only`：外部事实或系统快照，Agent 只能读取。
- `human_confirmed + read_only`：人工确认规则、参数和规范，适合规则表。
- `agent_generated + append_only`：Agent 生成的中间产物、候选结果和报告草稿，新增写入可由 runtime 直接放行。
- `mutable`：可修改对象，应谨慎使用，通常需要更强确认和审计。

如果结果需要人工确认后才能成为正式业务事实，不要直接建成普通可变事实对象。可以先建
`agent_generated + append_only` 的候选对象，再由单独的确认流程提升为 `human_confirmed`
或外部系统记录。

## Prompt 的使用边界

Prompt 适合：

- 告诉 Agent 标准流程。
- 说明工具选择偏好。
- 给复杂函数补充调用顺序和注意事项。
- 告诉模型何时应停止、何时应报告阻塞。

Prompt 不适合：

- 承担必须执行的校验。
- 代替参数 schema。
- 代替枚举、状态机和前置条件。
- 代替工具 handler 的错误返回。

如果一句 prompt 对正确性很关键，应优先考虑能否下沉到 schema、规则、函数返回或 validator。

## 从文档抽取的检查清单

阅读业务文档时，可以按下面顺序标注：

1. 用户会发起哪些任务？
2. 每个任务的入口对象是什么？
3. 任务中有哪些必须判断的关口？
4. 哪些关口有明确规则表或法规依据？
5. 哪些步骤会产生中间记录？
6. 哪些结果只是候选，哪些结果可以作为最终输出？
7. 哪些数据来自外部系统，哪些来自人工确认，哪些由 Agent 生成？
8. 哪些函数只是查询包装，哪些函数承载业务动作？
9. 哪些流程分支必须由工具错误或 validator 兜底？
10. 哪些开放性问题应留给 LLM 综合解释，而不是强行规则化？

完成初版 ontology 后，再做一次执行性收敛：

- 删除纯查询薄包装。
- 合并重复工具。
- 强化关口函数。
- 将 prompt 中的强约束下沉到确定性边界。
- 为核心任务加 prompts 和测试。

## 常见反模式

- 把文档目录结构直接翻译成 ontology 结构。
- 每个名词都建对象，但任务闭环不清楚。
- 每个对象都配一个 `get_xxx`，导致工具面过胖。
- 关键业务规则只写在 description 或 prompt 中。
- workflow 很完整，但函数只是 mock，导致执行结果看起来完成但没有真实业务判断。
- 把候选结果、人工确认结果和外部事实混成同一种可变对象。
- 让 LLM 自己记住“先审批再执行”“不通过则停止”等硬约束。

好的本体应该让 Agent 即使选错一步，也能被工具结果、校验和结构化状态拉回正确路径。
